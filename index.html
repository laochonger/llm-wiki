个人 Agent 的最后一块拼图

用 Claude Code ＋ Obsidian 构建 LLM-wiki · 一份高强度使用后的复盘

STATUS · DRAFTING DATE · 2026-05-15 AUTHOR · 老朱
TL;DR
文档散落在飞书、备忘录、浏览器收藏夹……记下来的速度永远追不上消费的速度。问题不在懒，在于记录链路的仪式感太重。受 Andrej Karpathy 的 LLM-wiki 启发，我用 Claude Code ＋ Obsidian ＋ Agentara 搭了一个个人知识库：三层架构、人机双读、零依赖，把加工放在写入时而不是查询时。这篇是高强度使用一个多月后的复盘 —— 为什么构建、怎么构建、在用它做什么，以及它跳不过的代价。

01引言

很长一段时间内，我的文档散落在各个平台。技术文章丢飞书收藏夹，灵感随手记在备忘录，B 站、小红书、浏览器收藏夹……虽然经常整理自己的内容到飞书 wiki，但是远远跟不上自己屯松果的速度，更别说积累自己的洞察与知识。虽然经常把文章丢给 agent 进行筛选和讨论，但常常是“阅后即焚”，没有留下什么有价值的思考，在“感觉要长脑子了”和“脑子是水做的吧”之间反复横跳。

直到我看到张咋啦的一条视频：“提升消费内容的仪式感，降低创作内容的仪式感”。原来问题不在我懒，而在于我的记录链路仪式感太重：开飞书文档要等加载、选目录、想标题、配图表，一套下来我已力竭；而信息流刷起来反而零摩擦。结果就是消费与收藏过剩、记录与创作贫瘠。

那么，有没有一种，既能满足囤积癖、边屯边整理，又能给到文章的价值判断，还能把学习过程都记录下来的方法呢？Andrej Karpathy：“有的兄弟，有的”。他提出了一个反 RAG 的思路——LLM Wiki：与其每次查询时把素材切片喂给模型，不如让 LLM 在写入时就把它整理成一个“持久、累积的产物”，交叉引用提前建立，矛盾提前标注，下次查询直接消费熟成知识。这非常符合直觉。

但批判的武器不能代替武器的批判，LLM-Wiki 还需要一个物理载体和一套操纵逻辑。在存储端，Obsidian 是天然的“血肉”。它的双链架构和纯文本底座，让知识不再是孤立的切片。这满足了对“拥有感”的追求，也为 LLM 提供了可以交叉引用的地址。而在执行端，coding agent，更具体地说是 Claude Code，则是我用于激活知识的“中枢神经”。Claude Code 执行任务的方式——“最简接口 + 渐进式加载 + 信任 LLM”与 LLM-Wiki 完全同构。当 Obsidian 提供 references/ 目录，Claude Code 并不需要扫描全库，而是通过 SKILL.md 与层层 CLAUDE.md 索引按需深读。同时，各类将 Claude Code 直连 IM 工具的项目也将对话的摩擦降到了最低。

实际上，LLM-wiki 的概念已经提出了一月有余，在 AI 圈，这应该是个不用学习的知识了。甚至由于这个概念过于直观，很多人在概念提出前就已经在这么实践了。但 天天上一当当当不一样 一千个人有一千个哈姆雷特，在较高强度的使用实践后进行复盘对我个人而言还是有一定价值的。

02为什么要构建个人知识库

1.1　集中记录：解决“和我之前看过的内容有什么关联吗”

知识库最朴素的价值，是把所有“值得回头查”的东西放进同一个仓库，用同一套链接关系组织。不再是“这条记到哪个 App 了”，而是“全在 wiki，搜一次就到”。我的具体做法是 sources/ 收原始素材、tech/ 收提炼后的概念、craft/ 收个人工程经验、qa/ 收沉淀过的问答……每一类都是“写一次、查无数次”的资产。

1.2　学习的复利：在降低摩擦和加速反馈之后

近两年很火的一句话“不断地记录自己，才能更好地进步”，粗看像鸡汤，但它精准指向输出端的复利曲线：每一次落笔都在给未来的自己积一份资产。低摩擦让人开始记，复利感才让人持续记，两把钥匙缺一不可。

网上很多的文章里都有同一个忧虑：长期依赖模型解释、模型生成、模型改写，自己原本能做的判断会退化，我也有同样的焦虑。所以我在 Claude Code 里挂了一个 hook，会话结束前，如果聊到值得记的技术点，自动整理进 wiki 的对应位置，并且值得深挖的点要记录到待办中定期推送给我。这是机制层面的“我”与“模型”的边界划分：模型可以加速思考，但“我自己沉淀过什么”必须在我自己的仓库里有据可查。三个月后要重新捡起这个话题时，不必从零开始“agent 一下”，而是直接跳回当时自己的总结。

1.3　Agent 的复利：让 Agent 越用越懂我

知识库不只是给人读的，它同时是 Agent 的长程记忆。每加一篇 sources/article、每多一条 craft/practice，下一次对话里 Agent 的回答就更贴我。这点和“自己复利”是同一份资产、两条收益曲线。同一份 Markdown（Anthropic：md 已死，html 当立），既给三个月后的自己看，也给下一次的 Agent 调用看，两条曲线一起涨。

1.4　看见自己：被动发现 blind spot

这是个意外的价值。我不是为它建 wiki，但 wiki 一旦运转起来，它就成了我每月对照自己的镜子。Lint 时浏览 sources/、craft/practices/、writing/ 的累积，一个月没写 craft，可能是最近停止深度思考；某条 idea 在 ideas/writing.md 挂了三个月没动，可能它本来就不该启动；最近 ingest 的文章全集中在 harness，可能视野收窄。

03如何构建

2.1　主流方案对比

构建个人知识库主流有三条路：

方案优势劣势
文档 + RAG检索语义化、对长文友好工程成本高、向量库维护、检索不可解释、人无法直观浏览
知识图谱关系结构化、推理能力强建模成本极高、个人场景过度工程
LLM-wiki（文档 + 双链 + 渐进式加载）人机双读、零依赖、Markdown 即结构依赖目录设计、召回可能弱于语义检索

LLM-wiki vs RAG 的差异，Karpathy 给了一个准确的概括：

“the wiki is a persistent, compounding artifact… the cross-references are already there. The contradictions have already been flagged.”
— Andrej Karpathy · LLM Wiki

RAG 的工作做在查询时（切片、向量化、召回、合成），LLM-wiki 的工作做在写入时（提炼、交叉引用、矛盾标注）。前者每次查都从原始素材重算一遍，后者每次写就给仓库加一次加工。

2.2　为什么选 LLM-wiki

中间层本身就是产品

RAG 不是“不给人看”，它完全可以在回答时附原文链接、跳回 source。但 RAG 的中间层（向量 chunk、embedding 索引）对人没有独立价值，人只读两端：原文和模型合成的答案，中间那层是机器看的。

LLM-wiki 不一样，中间层是 Markdown 实体，是我提炼过、组织过、互链过的版本，这一层既是模型查询时的索引，也是我平时打开 Obsidian 翻阅、写陈述、整理 outline 时的读物。同一份资产，人和模型消费的是同一个东西，只是入口不同。这一点对个人场景特别关键，因为我做这件事的边际收益不只是“以后好查”，更是“我每次写这层的时候自己也在巩固”。

渐进式加载够用，和 Claude Code Skill 同构

引言里点过：LLM-wiki 和 Claude Code 不是“借了工具”的关系，是同一套设计原则在不同层的应用。具体到 Skill 系统的对应：

Claude Code SkillLLM-wiki
SKILL.md 默认入会话index.md 默认入会话
references/ 按需读取tech/、craft/ 等 entity 按需深读
Skill 用路径互相引用entity 用 [[wikilink]] 互相引用

更深一层，这条设计原则可以总结为：最简接口（bash + filesystem 或 markdown + wikilink）、渐进式加载（默认入口 + 按需深读）、信任 LLM 自己组合（不预先建中间层抽象）。Claude Code Skill 是它在 agent 能力层的应用，LLM-wiki 是它在知识层的应用，Karpathy 和 Anthropic 独立得出了几乎相同的形状，是这条原则的横向印证。

落到个人场景甚至是具体的业务场景，这种“分层加按需”足够用。这点在去年做某取数项目时已经验证过了，只要上下文的注意力在“渐进式加载”（当时没有 skill 这个概念，但本质上是同一回事）下始终保持，那么就不要做向量化和图谱化。

零依赖

没有向量库、没有图数据库、没有部署服务。一个文件夹加 Git、Obsidian、任何 LLM CLI（我用 Claude Code）就能跑。哪天工具换了，资产还在 Markdown 里。

2.3　在 Claude Code 上的实操结构

Karpathy 在 gist 里给的是三层抽象架构（Raw sources / The wiki / The schema）。我在 Claude Code 里搭的这套 wiki，几乎是它的逐字落地：

下面是目录结构节选：

workspace/wikis/
├── index.md                # 入会话索引，列举各域 entity 概览
├── sources/                # 原始素材，read-only
│   ├── articles/           # 外部文章入库（带 frontmatter + summary）
│   └── notes/
├── tech/                   # 提炼后的技术概念
│   └── llm/concepts/       # KV cache、agent loop、harness 等
├── craft/practices/        # 我自己的工程经验（emerging→established→validated）
├── writing/                # 写作系统：ideas → drafting → published
│   ├── _index.md           # 文章库 + 类型分布
│   ├── ideas.md            # 管线（drafting 之后阶段）
│   └── articles/           # 每篇 = 一个素材总库（含所有平台版本）
├── ideas/                  # 想法捕获（用户主动输入，AI 不自动写入）
│   ├── writing.md          # 写作 idea
│   └── products.md         # 产品 idea
├── qa/                     # 问答沉淀
├── learning/               # 学习路径：python / algorithms / ai
├── logs/                   # 操作日志，按月归档
└── todo.md                 # 不进 Lint 的独立任务流

操作层面 Karpathy 给的三个动词（Ingest / Query / Lint）也直接映射进了我的 wiki skill：

  • 每篇外部素材走 Ingest 流程进 sources/ 并衍生 tech/ 实体；
  • 问答走 Query（先读 index.md，再回答）；
  • 定期 Lint 检查矛盾、断链、补 frontmatter 缺口。

每类目录有自己的 entity schema（type、frontmatter 字段、body 结构）。所有 entity 用 [[wikilink]] 互相引用，Obsidian 反向链接面板能直接看到“这个概念被谁引用过”。

嵌套 CLAUDE.md：规则跟数据走

到目前为止讲的 schema 都集中在顶层 wikis/CLAUDE.md。随着场景变多（阅读、写作、刷题、craft 提炼等），规则会散落到多处。我选的解决方式是嵌套 CLAUDE.md：每个高频域有自己的 CLAUDE.md，作为该域的 schema 加 workflow。顶层 wikis/CLAUDE.md 是总目录加全局 entity schema，子域 CLAUDE.md 是该域的局部 schema 加专属 workflow。

域子 CLAUDE.md 主要内容
writing/写作 workflow（draft → audit → publish）+ 引 [[craft/practices/laozhu-technical-writing-voice]] 拿风格规则
learning/algorithms/刷题 workflow（Open / In-Session / Close 三段）+ 选题类型判断 + AlgoProblem entity 局部覆盖
craft/craft 管理规则（confidence 三级升级流程 + 记录后必须告知用户的反馈循环）

这么做有四个好处：

  1. 规则跟数据就近。改刷题 workflow 直接改 learning/algorithms/CLAUDE.md，不用跳到 skill 目录。
  2. 单一入口。所有高频场景都走 wiki skill 触发，不需要每个场景建专门 skill，不会再有 algorithm-skill、writing-skill、fitness-skill 这种 skill 爆炸。
  3. Fractal 可扩展。新加场景（健身、阅读笔记、个人感悟等）只要建 域/CLAUDE.md，小幅修改 skill 和顶层 schema。
  4. 避免规则散落。原本可能进 memory feedback、craft、skill 三处的 workflow 类规则统一在子 CLAUDE.md。Memory feedback 缩到一行触发器（“做 X 时进 X 域 CLAUDE.md”），不再充当 workflow 容器，也就不会膨胀默认载入上下文。

2.4　三件套协同：Claude Code + Agentara + Obsidian

LLM-wiki 单独是个 markdown 文件夹，跑不起来。让它真正变成“持续累积的产物”需要三件配套：

Claude Code 提供 agent 能力底座

Skill 系统：wiki skill 定义 Ingest / Query / Lint 三个 workflow，识别关键词自动触发。SessionStart Hook：会话开头自动加载 memory/SOUL.md、memory/USER.md、memory/MEMORY.md，Agent 一开口就懂我。wikis/CLAUDE.md 作为 wiki 的 schema 定义（entity 类型、frontmatter、workflow），Agent 进入 wiki 操作前自动读。

Agentara（Lixin 大佬的一个 IM 连接 Claude Code 的项目）提供长程行为

三层 memory 自动累积，身份和偏好（SOUL / USER）加自动反馈（MEMORY）。定时推送 cronjob 主动推“今日 todo + 推荐阅读 + 该推进的事”，让 wiki 主动联系我而不是只在我打开时存在。

Obsidian 提供人侧界面

把 workspace/wikis/ 当 vault 打开。双链 [[wikilink]] 自动建反向引用面板。关系图谱可视化整个知识网络。

三件套各管一块：Claude Code 管 agent 怎么操作 wiki，Agentara 管长程偏好和主动推送，Obsidian 管我自己怎么读和导航。同一份 markdown，三个入口。

一句 prompt 搭起来

/skill-creator 帮我搭一个个人 LLM-wiki。先问我两件事：(1) 关心哪些领域（写作、读书、
刷题、AI、健身等，列 3-5 个）；(2) 已有的散落笔记位置和大致内容（Notion、飞书、
Obsidian 旧 vault、备忘录等）。基于我的实际需求设计目录树(不套模板)和 wikis/CLAUDE.md
三层 schema(参考 Karpathy LLM-wiki 和本文实践)。把已有素材迁入 sources/(read-only)
和对应域。装一个 wiki skill 定义 Ingest / Query / Lint workflow。配 SessionStart hook
自动加载 memory 三层(SOUL / USER / MEMORY)。整个目录做成 Obsidian-friendly：
[[wikilink]] 互引 + frontmatter 标准化。
搭完用我一篇真实素材跑一遍 Ingest 做 self-test。

最后把 workspace/wikis/ 加进 Obsidian vault 就完事了。

2.5　钩子：把记录变成默认动作

工具链再好，记不进去也没用。可以使用 Hook 降低创作摩擦：

  1. SessionStart。每次开会话自动加载 SOUL.md / USER.md / MEMORY.md 入上下文，Agent 记住使用习惯。
  2. 会话末提示。聊到值得入库的点，AI 主动问“这个要不要 Ingest”。
  3. Lint cron。每月一次自动 Lint，扫断链、查重、补缺，把变化提炼进 memory/USER.md。

降到“一句话即可入库”之后，很多以前会被忽略的小观察也开始进库。

04实践：我现在用它做什么

到这里都是结构和机制，但我的 LLM-wiki 上具体跑着什么活动？把我目前的高频用例拆出来，按工作流讲。

3.1　技术文章入库：一篇文章引发的四目录连锁反应

最朴素的入口，也是体感最强的。读到一篇文章（Karpathy 的 LLM-wiki gist、Cursor blog、某内部 harness 公开课），我直接丢链接，Agent 完成：价值判断，有价值的进行评分和入库 → 原文进 sources/articles/（read-only）→ 提炼 entity 进 tech/llm/concepts/ 或 tech/other/ → 扫描已有 concept 建反链、更新索引和 logs、标记阅读进度 → 结合我之前的记录和思考，返回给我我可能感兴趣的点进行对话和思考。

但 Ingest 的真正价值不止“收纳”，一篇文章从“我读过”升级到“我提炼了、跟踪了实现、反推到产品”，这才是 Ingest 在 LLM-wiki 范式下的真正含义。文章入库时 Agent 默认做四件事：

  1. 打分。按 frontmatter 的 rating 1-5 打分，依据是与我当前 focus 的相关度、新观点密度、可操作性。
  2. 找谬误。标出文章里我认同有问题或没充分论证的地方，作为后续 Q&A 种子。
  3. 找事实支撑。对文章关键事实主张做事实核查，补佐证或反例。
  4. 生成洞察。写一段不在原文里、属于“Agent 读后的推论”的内容。

如果当场没引发讨论，文章保留 status: unread。Agentara 每天 11:00 的推送会从 status: unread/reading 的文章里挑 2-3 篇带回来，附 Agent 写好的洞察让我继续讨论。讨论到我觉得“懂了”，状态升 read。

这一层让 ingest 不是单次动作，而是持续循环：先评估，再保留为 unread，定时推送带回继续讨论。

3.2　长文 / 书的协同精读：和模型一起读一本书

我读《置身事内》这种偏经济学 / 地方政府的内容时，由于自己没什么底子，光读容易“读过就忘”，或者“以为懂了其实没懂”。这正是 wiki 该接入的场景：模型补我跨域的认知缺口，wiki 替我把碎散的理解沉淀成可回查的结构。

具体设计是 6 段框架：核心议题、Q&A、洞察、原文总结、关键词、与后续章节的勾连。原稿存 sources/books/{slug}/（read-only），笔记反链。

跑下来体感是：读这种我没底子的领域，wiki 把“读一本书”从“看完合上书”变成“看完合上书还在仓库里继续生效”。一是我边读边提问边做事实核查，印象加深。二是之后想起这本书，跳回章节笔记，议题、问答、洞察、勾连全在，不用从零再读一遍。

3.3　写作管线：本文就是产物

writing/articles/{slug}.md 一个文件 = 一个内容单元的全平台素材库，长文版、小红书 9 宫格、B 站脚本、视频号短片在同一份 frontmatter 下分平台管理。本文就是这套结构跑出来的产物。当然，这其中还需要用到其他的 skill，但是 llm-wiki 让我积累的素材、思考更容易和写作进行链接，也更容易让我写的文章，和下次相同主题的创作进行关联。

3.4　GitHub 项目追踪：从“mark 一下”到“真吃透”

tech/repos/ 给每个值得吃透的项目建档：目标、技术栈、感兴趣的点、阅读进度。最近建档的有 hermes-agent、remote-claude、feishu-cli 等。每条 repo 页接 todo.md 的“吃透 [[X]]”一项，状态双向同步，“想看的项目”不再被信息流冲散。

3.5　工程经验提炼：emerging → validated 三级

craft/practices/ 收的是日常对话和 coding 中提炼出的判断，比如 [[craft/practices/wiki-structure-extraction]] 的“llm-wiki 目录切分原则”三问、[[craft/practices/local-cookie-as-auth-proxy]] 的浏览器自动化低成本鉴权范式、curl_cffi 提交脚本因 Cloudflare TLS 指纹升级导致失效改的修复过程。confidence 三级（emerging → established → validated）让我能区分“刚有这个想法”“印证过几次”“真的内化了”三种状态，避免把临时观察当结论。

3.6　反思与原则：动作之上的元规则

前面 5 条都是“做某件事”，但还有一类内容没有具体动作，纯粹是对世界、对自己的看法。reflections/principles.md 攒的就是这种东西。

这条线和前面用例的根本差异是，它是动作的指导逻辑，不是动作本身。但反思在传统笔记里很难体系化，因为条目之间彼此孤立。在 LLM-wiki 里有了双链以后，每条新原则可以在写的时候就标注“与既有原则呼应”。反向链接面板里立刻能看到“这条被谁引用、引用谁”，反思条目之间自然形成体系。甚至在中午、睡前等时间节点，定时推送自己记录的文字和感想来说服自己进行些许个身材管理和早睡劝导（你别说还挺有用）。

3.7　碎片捕获：todo / idea / 原则的三层入口

一闪而过的小观察要有零摩擦兜底。我用三层：

  • todo.md。一句 todo: 即可，连“是不是要做”都不判断，先扔进来。定期 Lint 清旧。
  • ideas/writing.md、ideas/products.md。想清楚一点的写作或产品念头。
  • reflections/principles.md。已经能抽象成原则的判断（详见 3.6）。

这一层里最有意思的是产品 idea 池。一些 idea 已经在碎片时间断断续续地走了十几轮的需求澄清，这是张咋啦那条“解自己痛点”最具体的实证，记录入口低摩擦之后，产品想法都开始有产出。

一条边界约束
ideas/ 不被 AI 自动写入，schema 写死了这条边界，AI 只在 idea 进入 drafting 阶段后才搬运。这条规则保护了 idea 池作为“用户人格”的纯度，它必须是我自己的输入。

3.8　算法刷题：调试过程比解法更值得沉淀

很不巧的是，从我工作开始，大模型刚好从能简单写代码进化到了能写好代码，所以工作后我的手动 coding 能力下降的厉害，就想着空闲时间刷刷 leetcode 保持手感。虽然刷题不是 wiki 最自然的用例，它的对话流和最终代码形态都跟“文章”“概念”差很远，但我还是给它单独建了一套 schema，原因是调试过程比解法本身更值得沉淀。Algo workflow 拆三段：Open 时把链接丢给 Agent，自动建账填 frontmatter；In-session 时所有“这个容器底层是什么数据结构”的 Q&A 由 Agent append 到该题“对话记录”段；Close 时一句“过了”，Agent 自动提炼语法点、追问归属哪个 pattern、建立 [[topics/X]] 反链。

刷题的副产物（语法点、模式归类、提交工具链 patch）以前都丢了或者需要编辑成题解，现在自动留下。

共性：把“本来就在做的事”接上沉淀

回头看 3.1–3.8 八条用例，有一个共同点：每条都是把“我本来就在做的事”接上 wiki 后变得有沉淀。我刷题本来就在刷，读文章本来就在读，记 idea 本来就要记，wiki 没有给我新任务，只是把这些动作的副产物自动归档。这点很重要，否则系统会变成额外负担，撑不过两周。

非主力但持续在用的还有几条：photography/street/ 上传一张街拍图自动分析拍法入档；qa/ 沉淀讨论结论。它们也符合上面这条共性，本来就在做，只是终于有了共同的去处。

05它带给了我什么

我把 llm-wiki 用成了学习工具，一边收集，一边学习，一边洞察，一边产出。这些事以前不是不能做，是有了知识容器之后反馈变快了。想象一下，在你的学生时代，有一个 24 小时在线的老师，能在分钟级别内回答你方方面面问题，附带推导过程和出处，最后帮你记录好讨论过程，觉得你掌握的不好还会定期拉着你复习，这得少掉多少头发啊。

4.1　更多的 Idea

记录方式简化之后，最直接的变化是愿意记了。以前一个想法冒出来，掂量一下要不要记录，多半放弃。现在直接 /wiki todo 一句话回车，一周就有四条 ideas/writing.md 和两条 ideas/products.md，都是以前会被信息流冲散的种子。

4.2　碎片时间做完整的事

这是我从张咋啦那条小红书拿到的最大红利。以前碎片时间只能用来消费，刷刷信息流、看半个视频。现在碎片时间能做完整的小动作：把一篇刚读完的文章 Ingest 进 sources/、把一个突然想通的技术点写成 craft/practice、把一个问题和结论沉淀成 qa/。每个动作独立完整，可叠加，“完整”的颗粒度变小了。

4.3　缓解 FOMO

技术圈每天有一百条“必须知道”的更新。FOMO 的根源不是信息多，是缺一个“这条信息是否属于我现在的主线”的判断器。LLM-Wiki 是这个判断器：

  • 属于已有 concept，Ingest，扩充。
  • 属于新方向，进 ideas/learning.md 排队。
  • 都不沾，安心略过。

有了归处，信息就不再是“错过就亏”的稀缺品，按需调用即可，FOMO 自然降下来。

06它的代价：不是银弹

LLM-wiki 这套东西概念上的确干净，三层架构、人机双读、零依赖、写入侧摊销，每一条 Karpathy 在 gist 里都说得明明白白。但概念归概念，搭起来其实不是这么回事，至少有两个代价跳不过去。

5.1　前期投入是负收益

Schema 没有最佳范式，目录怎么切、entity 字段怎么定、Ingest workflow 该长什么样、sources/ 要不要 read-only、ideas/ 要不要禁用 AI 写入，全都得边用边做。至少第一周收益曲线在水平线下，你在“建仓库”而不是“用仓库”。

5.2　它不会帮你学习和思考

它并没有完全解决收藏了不看的问题。它只是把学习的成本降到最低、把反馈的速度拉到最高，但能不能学进去，最后还是看你自己愿不愿意打开它。

5.3　不是百分之百会被调用

Agent 不一定每次都查 wiki。Implicit Query 规则会漏读，已加载的 index.md 信息也未必足以触发它去深读具体 entity，更不用说它有时直接凭模型的内置能力或者调用其他 skill 作答，wiki 那一份被直接绕开。不过等你用的多了，agent 记住了你的习惯以后，这也不会是个大问题。

07结尾

Karpathy 在那条 gist 末尾留了一句很关键的话：这套模式有意保持抽象，使用者按自己需求改造，而不是僵化套用规范。你不需要一开始就把所有目录、所有 entity 类型设计完整，从 sources/ 一篇文章加一条 craft/practice 开始，剩下的让它在 Lint 中长出来。

但说实话，这一整套东西，它有效的部分几乎全部在于“学习”这件事本身，好像并不解决什么实际问题。某种程度上，可能只是像在学生时代那样，用“假装努力”来对抗焦虑。我清楚地知道，我用它记下来的概念，三个月后多半也会忘，模型迭代两轮，累积的资产可能就过时。

但我们应当想象，西西弗斯是幸福的。
