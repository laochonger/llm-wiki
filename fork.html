01引言：小径分叉的花园

我有个老毛病，一个问题问着问着，更多的问题就涌了上来，必须赶在瞬时记忆消失之前噼里啪啦一顿操作，否则忘了就不用学了。其实问题多倒也不愁，这是好事呀，愁的是它们一批批地、三五成群地涌上来，却没有一个很好的方式来处理。于是，花园里分叉的小径越来越多，状态越来越混乱，手里好几份报告就这么难产。不是不写，是要缓写，慢写，有规划地写，是注意力被发散的好奇心扯散，主线迟迟收不了口。

在这之前，我解决这种发散有几条出路，但每条都差了那么点意思：

  • n个问题打包成一段问出去，回答又长又糊，哪个都没说透；
  • 一个个串行问，等问到第三个，后面的早忘了。到最后，即使我的脑子还记得住，模型也开始大海捞针了；
  • 丢个 subagent 去查，它上下文是空的，得重新喂背景，上下文不全，能用的工具也有限；
  • 起 /btw，不能用工具，还无法自动回传；
  • 起个 claude -p，agent同样要重新交代，还落在另一个窗口；
  • 大力出奇迹，/branch，但/了一次branch，仍有无数的branch要/。

而这个问题，也是我开始着手构建自己的工具的原因之一。但人是一直踏入同一条河流的，老毛病犯了，工具也边写边发散，边发散边忘，根本无法收敛。

还好， /fork 大人来了，我有救了。按官方文档，它是一个继承了到目前为止整段对话的 subagent，最终会把结果回流，主窗口保持干净。直观上，它非常适合做非主线杂活，保证我阅读时的好奇心不消散。但我叒踏入了同一条河流，单是看一眼官方文档，就能勾出n个疑问：它如何保证命中缓存？一次能起几个 fork？fork 能不能套娃其他指令？返回的东西长什么样？和 /branch差别在哪？和之前的subagent差在哪？By the way，它和 /btw 指令有啥区别？

而既然要更加了解这个工具，这个工具直观上又最适合做非主线杂活，不如让英雄好汉查英雄好汉，派 fork 去查 fork。这篇文章里几乎每一个结论，都是这么来的。一个对话之内，我一直 /fork，把实验做完、把资料查完，主线只负责和我一起，把回流的结论逐批地核对（如果觉得结果需要核查，主agent可以auto-fork）、落wiki目录（内部文档），最后收拢成文，走出小径分叉的花园。

02一、/fork 是什么

全量上下文 + 隔离、并行、异步和回流。 按 Claude Code 官方文档，/fork 是一个继承了到目前为止整段对话的 subagent：它看到和主会话一模一样的 system prompt、工具、模型、消息历史，所以你甩给它一个旁支任务，不用重新解释背景；它自己的工具调用留在自己的上下文里，只有最终结果回流，主窗口保持干净。要紧的是它能一次起很多个：n 个问题 n 个 fork，想到就甩一个，各跑各的，主线不停，追问也只是再起一个。零碎时间里把发散的问题并行扔出去，不用守着等，主线那份报告反而能一口气写完。

它怎么触发，那个开关到底改什么。 fork 不用手动开——从 v2.1.161 起 /fork 命令默认就能用。触发 fork 主要三条路：

  1. 你直接打 /fork 命令+指令，带不带 CLAUDE_CODE_FORK_SUBAGENT 都行；
  2. 设了那个CLAUDE_CODE_FORK_SUBAGENT=1后，模型本来要起冷启subagent（比如general-purpose）的地方会自动改起 fork（fork 本身也是 subagent，严格说是换一种 subagent 的起法）；
  3. skill 里配 context: fork。

所以那个环境变量改的不是"能不能 fork"，而是"模型默认派生subagent时要不要走 fork"，实际上，这个行为是可以被打破的，第二节再细嗦，细嗦不是胡说。

落盘是一个指针。 fork 的磁盘转录第一个节点，不是把父对话拷一份，而是一个指针 fork-context-ref，只存 parentSessionId、parentLastUuid、contextLength 三个字段。完整 prompt 由 harness 顺着这个指针在发请求时重建，重建出的前缀和父会话逐某大厂一致。这就是它命中父缓存的原因。

回流走一条合成的 user 消息。 这是 fork 和 subagent 最容易混的地方。一般来说，subagent 回流是一个 tool_result 块，靠 tool_use_id 和主 agent 的 tool_use 配对，同步、卡在那一轮里。fork 不一样，它回流落盘的是 {"type":"user","origin":{"kind":"task-notification"}}：不和任何 tool_use 配对，异步。fork 在后台跑完，harness 把这条 system 来源的 user turn 注入进来、重新唤醒主循环。信封里值钱的是 <result>（唯一进主 context 的东西）和 <usage>，外加一个 <output-file> 指向完整转录，是事后唯一能审计的落点。

回流的结论 Microcompact 擦不掉。 Claude Code 每轮跑一个 Microcompact（内部文档），会擦掉旧的 tool_result，但从不动 user 消息。而 fork 的 <result> 恰好落地成 user 消息，于是它的结论永久钉在主 prefix 里、擦不掉；同样的活你 inline 自己干，产生的是一堆会被擦掉的 tool_result。所以 fork 干活时主线最干净（脏活锁在它那边），但它留下的结论最赖着不走。

它的成本结构。 fork 顺着指针重建父前缀，第一发就热命中缓存，继承一整段历史这件事本身几乎不额外花钱。fork 在继承来的上下文里反复读、调工具、推理，单个能跑到几十万 token，过程全隔离，回流进主线的只各几百字的 <result>。所以 fork 花的，不是"把上下文重背一遍"（那部分被缓存接住了），而是它自己跑那一大圈、外加主线把回流结论收拢整合的那点开销。

03二、它和别的"委派"方式差在哪

把活交给另一个 agent 去跑，Claude Code 里有好几种方式，差别就在三个轴：是否继承上下文、有没有工具、结果回不回到主线。先摆一张表，下面逐个对比。

fork 其实就是一种 subagent。 官方 sub-agents 文档原话是：

“a fork is a subagent that inherits the entire conversation so far… drops the input isolation that subagents otherwise provide.”

所以subagent 下分两支，冷启 subagent（Agent 工具起的那种：context 空、得重喂背景、工具可限权、适合做限权的只读 worker（可配 disallowedTools）、干完回一段摘要）和 fork（带着你全部上下文来那种）

fork 和冷启 subagent 区别：

  1. 上下文全继承、不漏思考信息。fork 看到主会话至此的全部 system prompt / 工具 / 消息历史，连你中间那些推理都在场；冷启 subagent 只拿一段手写的 task 摘要，思考信息全丢了。
  2. background 命中父缓存。fork 顺指针重建父前缀，符合前缀缓存规则；冷启 subagent 从零起、read=0。
  3. /fork 体感异步。你打 /fork 是甩出去就走、主线继续跑；它在后台排队/执行、跑完异步回流（主 agent 正忙时就 hold 在队列里）。

这三条合起来才是 fork 的价值、也是它和冷启 subagent 的真区别。至于"能不能再委派"是另一层能力差：fork 继承了 Agent 工具，所以是可以再起 subagent的、冷启 subagent 的 Agent 工具被剔了所以不能。

哪次subagent委派会变成 fork。 派发逻辑只看派发subagent时，是否填了subagent_type，缺了走 fork 分支拷父对话，填了（比如 general-purpose）就起冷启 subagent。所以第一节那两条触发路的差别就在这：

  • /fork 命令带不带 env 都起真 fork；
  • 主agent的 auto-fork 需要满足：设置 CLAUDE_CODE_FORK_SUBAGENT=1 && 调用subagent工具时，subagent_type 缺省。所以CLAUDE_CODE_FORK_SUBAGENT=1 设置的是默认用fork的方式派发subagent，而非强制。所以即使设置了环境变量，也可以启动别的类型的subagent，比如直接说“派生一个 genreral-purpose subagent”。

顺带分清两个常被混的字段：

fork vs /branch。 /branch 也带着满上下文，但它是"搬家"不是"派活"：从当前这点岔出一条独立的、能继续聊下去的平行会话，控制权整个搬过去，结果不回流到主线。所以 fork 和 branch 在缓存上没本质差别，真区别只在控制流：

  • fork 派出去、跑完把 <result> 回流给主线；
  • branch 是你自己走进那条线、不回流。

fork vs /btw。 /btw 是"看着全文问一句、不许动手"的只读旁问：它带着满上下文，却既不使用工具、也不把这轮写回缓存、更不落转录，问完即焚，所以特别便宜（满 context 全走 cache_read、新问题只多那几十 token）。

但它怎么做到不使用工具？把 tools 从请求里删去肯定不合理吧？/fork 去grep 源码后，发现是三层叠出来的：

  • wire 里的 tools 数组原样保留（前缀不变、缓存命中）
  • harness 的权限层 canUseTool 直接返回 deny（"Side questions cannot use tools"）
  • prompt 层再补一句"你没有任何工具可用"。

这三层正好把软硬分了出来：

  • canUseTool 那道 deny 是硬的，模型再想调也执行不了；
  • 而 fork 的 boilerplate 里 Do NOT spawn subagents 只是 prompt 文本，是软的。我要求一个 fork 去调 Agent 工具，它真起了一个子 agent。

再往下看一层，/btw 还不是个专门机制：源码里 querySource 有 31 个取值（compact、generate_session_title、extract_memories…），side_question 即 /btw 只是其一，这一族廉价一次性查询都走 maxTurns:1 + skipCacheWrite + skipTranscript 同一套，读缓存但不写回、也不落盘，这也是为什么当我想通过本机上的转录来验证结论时，翻遍目录也找不到。

fork vs claude -p。 claude -p是 Claude Code 的 headless 入口，从 shell 起一个独立的顶层会话，跑完把结果打到 stdout。它归在 claude 那一档主 agent 里，不是狭义上的 subagent。正因如此，它看着也"带工具、能跑活"，却不在主会话里：没有 parent、fresh context、背景得重新交代。

  • fork 是会话内派出、回流给 parent；
  • claude -p 是另起一个进程世界。

fork 自己不能再 fork，但它经 Bash 起一个 claude -p，就跨过了进程边界。

Harness的许多设计被同一件事拽着：保住前缀缓存。 缓存按严格前缀算（顺序 tools、system、messages），改 tools 那一段就让后面整条作废（官方原话 "Modifying tool definitions invalidates the entire cache"）。回头看，好几处都在避免踩到这条红线：

  • /btw 死活不删 tools、改在prompt中提示 + harness 权限层 deny；
  • fork 用指针逐某大厂重建父前缀、好命中，而且不删subagent派发工具；
  • branch 直接复用父会话那条还热着的缓存；
  • 连 tool-search 加载新工具，也是把 tool_reference 塞进 messages 尾、不动前缀。

不动工具名单怎么让它动态加载工具呢？各家有不同的方案，可以参考 Agent 工具发现：从上下文膨胀到按需加载

读 harness 的设计，"这一步是不是在护缓存"几乎是把万能钥匙。

04三、它能起多少、能派生什么

并发上限和排队。 让 agent 去 grep 本机 claude.exe（v2.1.165）读出来的是源码里那个 XUO 函数，min(16, max(2, 核数 − 2))，传进 CPU 核数，我这台 12 核代入是 10。（同一个上限也管着 Workflow 里 agent() 的并发，≤16，两处共用。）实测两批交叉验证：13 个 fork 执行窗口峰值 6，25 个 fork 峰值贴到 9、排队从 10.7 拖到 47.7 秒。超上限是排队、不是被拒。

接收是批量，不是串行。 回流之后主 agent 也不是一个个处理：fork 在主 agent 说话时跑完的，通知先攒着，等这轮一说完，harness 把攒着的一次性打包塞进下一轮（实测时最多的一批 8 个一起进）。所以 n 个 fork 不会摊成 n 个串行轮，唯一躲不掉的那点串行，是"把 n 个结果放一起综合"得在一个 context 里做，花一轮生成，不是 fork 跑的那几分钟。

fork能套娃吗？先说明两点：

  • 冷启 subagent 不能再 spawn：构造它的工具集时 Agent 工具被剔掉了，Explore/Plan 的注册项也明写 "All tools except Agent"。实测起 general-purpose subagent，自报工具里确实没有 Agent。
  • fork 不能再直接 fork：不是删工具，而是"省略 subagent_type → 起 fork"那条派发路在 fork 内部不再触发（fork-context-ref 继承深度=1、只给一层），所以 fork 内即便省略 subagent_type，也只落成冷启 subagent、不是嵌套 fork。

“A fork cannot spawn further forks.”

但 fork 手里仍是全量工具，冷启subagent、claude -p、workflow 它都能发起。派发fork时，prompt里的 Do NOT spawn subagents 只是可穿透的软禁、不是第二道硬闸。这条硬限不是"不许递归"、也不是"不许更多 agent"，它锁的是 fork-context-ref 那套继承机制的深度，只给你一层。fork 自己生不出 fork，但它经 Bash 起的 claude -p 是个独立顶层进程，那个进程既能再 fork、又能再起 claude -p，理论上任意深。换句话说，想堆更深，换条机制（走进程边界）就能绕开这堵墙，只不过要交过路费（每过一道边界丢一次上下文继承、退回冷启）。

05四、这几个命令我平时怎么用

上面那张对比表落到日常，大致这么分工：

  • 要把一坨一次性检索或日志关在外面、只取结论，（大多数情况下主agent会自动指定）用 冷启subagent（顺便能限权，做个只读的安全 worker）；
  • 要并行追一堆发散的问题、还想要结论回来，用 /fork；
  • 要从当前这点岔一条能继续聊下去的平行会话，用 /branch；
  • 只想看着全文问一句、不动手不污染主线，用 /btw（但我很少使用，也许问题能在单次问答中满足，但好奇心不行）；
  • 要在脚本里批量跑、headless 自动化，用 claude -p。比如使用skill的评测循环就是 spawn 一批 claude -p 拿同一个 query 反复跑、测描述稳不稳，再配两个 subagent 打分；
  • 不过任务一跑长，进程偶尔会半路歇火；这时我在外头加个 /goal 当监工，让它一直游到海水变蓝。但/goal和/effort utlracode(workflow)都基于一个前提：你的问题已经基本被定义清楚了。

但/fork也有一些注意点：

  • 手动 fork 带不上图。 图粘进 /fork 命令本身 → 记成 attachment 文件引用,不展开成视觉块 → fork 只拿到 [Image #N] 占位符 → 看不到(除非它自己去 Read 文件)。
  • 产出结果不会被 Microcompact。 fork 的结果落成 user message、Microcompact 不擦，永久钉在主 prefix；要可擦的用 inline subagent（留下能被后续清理的 tool_result）。但是返回的结果本来就不多，不擦更符合预期。
  • fork 其实不便宜。 带了全量上下文，思考的更加全面，一个 fork 烧几万到上百万 token，回主线的只有几百字 <result>。它省的是"主线干净"、不是 token。

另外，这趟用/fork调研/fork中最该记的，也许是我栽的几次车，它们划出了"在一个会对我撒谎的系统里怎么求真"的边界。我的调研流程里，在每次批量/fork的答案返回以后，主 agent 对需要核验的结论，从外部进行核对（读 transcript 的 token、时间戳、.meta.json、grep 二进制），而经过验证或者被推翻的结果，都会按照落盘到wiki里的qa树上。

但这个方式也有它够不着的盲区，它能戳穿 agent 的自报，却仍然戳不穿"提问者自己埋进去的前提"。比如"自动 fork 没复现"那次，真凶是agent做实验时的prompt里填了 general-purpose，用自己的参数把结论否了，即使开个 subagent 去查也是同一个盲区。这种"自己的动作污染了结果"的 confound，agent难以察觉。那一层暂时只能靠我自己的直觉和判断，这也是为什么定论最后还得过人的判断。所以我喜欢使用/fork的另一个原因是，它的返回速度和返回的信息量，刚好是我的能力能cover住的。

那有聪明的同学就要问了，同学同学，你为什么不开deep-research呢？ 那我问你，你开智驾模式上班手为什么不离开方向盘(bushi)？ deep-research（连同 workflow 模式）把验证这步交给里头几个对抗 subagent 互相找茬，机器查机器，看起来很美好。 但就目前我自己的使用实践看，调研可能使用deep-research，但deep-research用来调研有点不太可能（诶我的头怎么尖尖的），更靠得住的还是人的判断：模型到底有没有 grep 源码、该疑哪条自报、哪个反例不能当噪声扔，这些是那几个验证 subagent 难以做到的。即使验证思路能落到skill，但“直觉”却难以复刻（也许是我经验不足，难以直接从返回的海量信息中揪出错漏之处🤫）。当然，这也许是模型在长上下文时还不足够强，但我无法判断模型的边界到底在哪里，概率能否将幻觉变成真的直觉。 而fork 把人留在裁判位上，一批一批地积累qa树，每一批问题与结论，我都能注意到。 什么叫自动挡油门直接踩到底的操控感啊（战术后仰）。

06五、/fork带给我什么

追问的摩擦减少了。 回到开头那个老毛病。以前那些发散的问题是负担，因为问一个的代价不低：要么打断主线、要么另开窗口、要么忍着。fork 把这个代价压到几乎为零，想到就甩一个，主线不停。代价一降，好奇心就不再被压着，反而越用越多，越来越转换为稳定的求知欲：一个问题勾出五个，五个再各自勾更多，就像是这篇文章，用100多个fork去调研fork。学习这件事的摩擦，被又往下压了一档。

妈妈再也不用担心我边写作边提问啦。写这篇写到一半，我又想顺手核几个点，fork 到底跑不跑得起 claude -p？能不能发起 workflow？agentType的类型有多少种？手动/fork+图片为什么无法生效？这都是写着写着冒出来的岔路。换以前，要么停下来去开几个新对话查实验、打断这篇，要么记个 TODO 等它凉。这次我直接甩 fork 去验，不用起新对话，而session的上下文依旧健康。

好奇心照样发散，但这次回答收敛住了。 /fork让我在提问->纠正->落盘->产出的流程在一个主对话中闭环了。我的使用方式是，想到的问题就开个 fork 去问，回流时人和主线一起查验，并结合LLM-wiki落盘，记录、合并、纠正问答树。像git一样，主线干净，旁路清晰。

回到那个学习工具。"读的时候并行涌现一堆问题、别让它们排队冷掉、也别把主线拖垮"，正是我在做的学习工具想解的问题之一。现在 harness 层的 /fork 把它直接解了一半，idea也因此有了新的实现方向，而不是困在实现基础功能上。

07结尾：当我谈论/fork时，我在谈论什么

从学生时代起，我对自己的定位就很清晰：好奇心有余而求知欲不足。对于Agent的探索也是如此，虽然一直都保持着好奇，但经常由于畏难而认知滞后。本质上来说，其实是不够聪明，故而探究事物时畏惧复杂逻辑。但又有点小聪明，会去判断探究到什么程度时，边际效益就衰减的厉害。考试只需要每一科都考的还行，就能上还行的学校；代码能力和沟通能力没有明显的短板，就能来所谓的大厂拧螺丝。

但 Agent 时代，其实给每个人都外挂了一个好奇心满足系统，好奇心被满足的越来越快，求知欲也就越来越趋于稳定。当我在谈论/fork时，也许想谈论的是，它如何在Agent时代再次降低了推石头的摩擦，让我可以把“边际效益”这种理性的算计暂时抛在脑后。唯一的问题，可能是比较影响睡眠，我缺的睡眠这块谁来给我补啊.jpg。
